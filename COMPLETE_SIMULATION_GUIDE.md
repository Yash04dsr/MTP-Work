## 1. The Physical Problem

### What are we simulating?

A **T-junction pipe mixer** where pure hydrogen (H2) is injected from a small branch pipe into a large main pipe carrying pure methane (CH4), at high pressure (69 bar). This models hydrogen blending into existing natural gas pipelines.

### Why does it matter?

- H2 is **8x lighter** than CH4 (density: 0.082 vs 0.668 kg/m3 at standard conditions).
- Buoyancy drives the H2 upward, causing **stratification** instead of uniform mixing.
- Poor mixing leads to: (a) hydrogen embrittlement at the pipe crown, (b) Wobbe Index drift causing burner instability, (c) metering errors.
- We want to find the **optimal injection geometry** that maximises mixing quality while minimising pumping cost (pressure drop).

### The two objectives (what we optimise)

| Metric | Symbol | Definition | Goal |
|--------|--------|------------|------|
| Mixing quality | **CoV** | Standard deviation of H2 mass fraction at outlet / mean H2 at outlet | Minimise (0 = perfect) |
| Pumping cost | **\|Î”P\|** | LP-filtered gauge pressure drop from inlet to outlet | Minimise |

### The three design variables

| Variable | Symbol | Range | What it controls |
|----------|--------|-------|------------------|
| Injection angle | Î± | 30 deg, 90 deg | Tilt of the branch pipe relative to the main pipe |
| Diameter ratio | d/D | 0.15 - 0.45 | Size of the branch relative to the main pipe |
| Velocity ratio | VR | 0.69 - 5.84 | How fast H2 enters relative to CH4 flow |

### The four campaigns (40 total simulations)

| Campaign | Angle | Injection side | Cases |
|----------|-------|---------------|-------|
| 90 deg top | 90 deg | Top (branch at y = +R1) | 10 cases |
| 30 deg top | 30 deg | Top | 10 cases |
| 90 deg bottom | 90 deg | Bottom (branch at y = -R1) | 10 cases |
| 30 deg bottom | 30 deg | Bottom | 10 cases |

## 3. Geometry Generation

### The script: `generateSTL.py`

This Python script creates STL (surface mesh) files that define the T-junction geometry. snappyHexMesh uses these to carve a body-fitted mesh.

### How it works step by step

1. **Read parameters from environment variables:**
   - `D1 = 0.460 m` (main pipe diameter, fixed)
   - `D2` = varies per case (branch diameter, from d/D * D1)
   - `L_MAIN = 6.900 m` (main pipe total length)
   - `Z_JCT = 2.300 m` (where the branch meets the main pipe)
   - `L_BRANCH = max(1.380, R1 + 12*D2)` (branch length, at least 12 diameters for flow development)
   - `ALPHA_DEG` = injection angle (90 for perpendicular, 30 for tilted)
   - `BOTTOM_INJECT` = 0 or 1 (whether branch is at y = +R1 or y = -R1)

2. **Create the main pipe wall:**
   - Parameterised as a cylinder in (theta, z) space.
   - Discretised into N_CIRC = 96 circumferential points x N_AXIAL_MAIN axial points.
   - A rectangular hole is cut where the branch intersects.

3. **Compute the intersection curve:**
   - The exact curve where the branch cylinder pierces the main cylinder.
   - This is computed analytically at 96 circumferential points.

4. **Zipper triangulation:**
   - Connects the rectangular hole boundary on the main pipe to the intersection curve.
   - This is the critical step for water-tightness -- the hole edges and branch base share the EXACT same vertices.

5. **Create the branch pipe wall:**
   - Linear interpolation from the intersection curve up to the branch inlet cap.

6. **Create end caps:**
   - `main_inlet.stl`: disk at z = 0
   - `outlet.stl`: disk at z = L_MAIN
   - `branch_inlet.stl`: disk at the top of the branch

7. **Water-tightness check:**
   - Computes signed volume using the divergence theorem.
   - Compares to the known analytical volume (main cylinder + branch cylinder - overlap).
   - Aborts if the ratio deviates > 3%.

8. **Output files** (in `constant/triSurface/`):
   - `wall.stl` - all pipe walls
   - `main_inlet.stl` - inlet cap
   - `outlet.stl` - outlet cap
   - `branch_inlet.stl` - branch cap
   - `tJunction.stl` - combined (for visualisation)

### For tilted branches (30 deg)

The `ALPHA_DEG` parameter tilts the branch axis. Instead of pointing straight up (+y), the branch axis tilts in the y-z plane. The intersection curve changes accordingly, and the branch wall is generated along the tilted axis. The tilted geometry produces a longer, streamwise-aligned shear layer that improves mixing.

### For bottom injection

When `BOTTOM_INJECT=1`, the branch is placed at y = -R1 instead of y = +R1. The branch points downward. H2 must now rise against gravity through the entire pipe cross-section.

### The clip_stls.py script

Since we use a **half-domain** (symmetry at x=0), the STL files need to be clipped to x >= 0. This script:
- Reads each STL file
- For each triangle: if all 3 vertices have x >= 0, keep it; if all have x < 0, discard it; if partially crossing, clip at x=0 by interpolating new vertices
- Writes the clipped STL back

Without this step, snappyHexMesh silently drops patches whose STL lies outside the background mesh.

## 4. Meshing Pipeline

The mesh is built in 4 steps:

### Step 1: `surfaceFeatureExtract`

```bash
surfaceFeatureExtract
```

- Reads `system/surfaceFeatureExtractDict`
- Extracts sharp edges from each STL file at 150-degree included angle
- Outputs `.eMesh` files in `constant/triSurface/`
- Purpose: snappyHexMesh uses these to snap mesh points onto sharp geometric features (pipe ends, junction intersection line)

### Step 2: `blockMesh`

```bash
blockMesh
```

- Reads `system/blockMeshDict`
- Creates a uniform hexahedral **background mesh** that encloses the entire geometry
- Our settings:
  - Half-domain: x in [0, 0.30], y in [-0.30, 1.50], z in [-0.05, 6.95]
  - Cell size: 30 mm uniform
  - Grid: 10 x 60 x 232 = **139,200 cells**
  - Patches: `sym` on the x=0 face, `allBoundary` on everything else

### Step 3: `snappyHexMesh -overwrite`

```bash
snappyHexMesh -overwrite
```

This is the main meshing tool. It reads `system/snappyHexMeshDict` and does three things:

**Phase 1: Castellated mesh (cell removal + refinement)**
- Removes all background cells that lie outside the T-junction STL surfaces
- Refines cells near important features:
  - `junctionRefine`: a sphere centred at the junction (radius ~ 3*D2), refined to **level 2** (cell size = 30/4 = 7.5 mm). This is where the jet enters and the most complex flow physics happen.
  - `jetRefine`: a cylinder in the downstream mixing region, refined to **level 1** (cell size = 15 mm). This captures the decaying vortex structures.
  - Wall surfaces: refined to level 1
  - `locationInMesh`: a point inside the domain (0.077, 0.011, 1.111) that tells snappyHexMesh which side of the STL is "inside"

**Phase 2: Snapping**
- Moves mesh vertices onto the STL surface geometry
- Uses the `.eMesh` feature files to preserve sharp edges
- Settings: 3 smooth iterations, tolerance 2.0, 50 solve iterations

**Phase 3: Layer addition**
- Adds **3 prismatic boundary layers** on the `wall` patch
- Expansion ratio: 1.2 (each layer is 1.2x thicker than the previous)
- Final layer thickness: 0.5 (relative to the near-wall cell)
- Purpose: resolves the boundary layer for the wall-function approach (target y+ = 30-80)

**Result**: ~445-480k cells (varies slightly per case due to different branch diameters)

### Step 4: `checkMesh`

```bash
checkMesh
```

Validates mesh quality:
- Max non-orthogonality: < 65 deg (ours: ~49 deg)
- Max skewness: < 4 (ours: passes)
- Min cell volume: > 0 (no negative-volume cells)
- Layer coverage: 99.6%
- If checkMesh finds fatal errors, the case is aborted

### Mesh reuse (the slice optimisation)

The DoE uses a **sliced LHS** where 2 cases share each d/D value. Since the geometry only changes with d/D (not VR or HBR), cases in the same slice can **reuse the same mesh**:

1. Case with lower ID in a slice builds the mesh from scratch
2. Case with higher ID copies `constant/polyMesh/` from the first case
3. Only the boundary conditions in `0/` differ

This cuts meshing time in half (5 mesh builds instead of 10).

## 5. Boundary Conditions

Each field has a boundary condition on each of the 5 patches. Here is every field explained:

### U (Velocity)

| Patch | BC Type | What it does |
|-------|---------|-------------|
| `main_inlet` | `codedFixedValue` (mainInletPowerLaw) | 1/7th power-law turbulent pipe profile: `u(r) = u_max * (1 - (2r/D)^2)^(1/7)`. Direction: (0, 0, u) = flow in +z direction. This mimics fully-developed turbulent flow without a precursor simulation. |
| `branch_inlet` | `codedFixedValue` (branchInletPowerLaw) | Same 1/7th power-law, centred on the branch axis. Direction: (0, -u, 0) = injection into the main pipe (downward for top injection). |
| `outlet` | `pressureInletOutletVelocity` | Zero-gradient on outflow (velocity is free to adjust). If backflow occurs, applies a fixed value to prevent instability. |
| `wall` | `noSlip` | Zero velocity at all walls. |
| `sym` | `symmetryPlane` | Mirror: normal component = 0, tangential unchanged. |

**The @TOKEN@ system**: The actual values of `@UMAIN@`, `@UBRANCH@`, `@D1@`, `@D2@`, `@ZJCT@` are placeholders that get replaced per-case by `stamp_cases.py`. For example, if VR = 3.81 and U_main = 10 m/s, then `@UBRANCH@` = 38.1 m/s.

### p_rgh (Pressure minus hydrostatic head) -- THE SOLVED PRESSURE

| Patch | BC Type | What it does |
|-------|---------|-------------|
| `main_inlet` | `fixedFluxPressure` | Adjusts pressure gradient to match the specified velocity BC (doesn't fix pressure itself). |
| `branch_inlet` | `fixedFluxPressure` | Same as above. |
| `outlet` | `prghPressure` | Fixes the reference absolute pressure at 6.9 MPa (69 bar). The solver computes p_rgh from this. |
| `wall` | `fixedFluxPressure` | Ensures no flow through walls. |
| `sym` | `symmetryPlane` | Mirror. |

### p (Absolute pressure)

All patches: `calculated`. This field is derived from p_rgh + rho*g*h by the solver. It is not directly solved.

### T (Temperature)

All inlets: `fixedValue` at 288 K. Wall: `zeroGradient` (adiabatic). **Energy equation is disabled** (maxIter = 0 in fvSolution), so this stays at 288 K everywhere. This is the isothermal approximation -- valid because both streams enter at the same temperature and there's no combustion.

### H2 (Hydrogen mass fraction)

| Patch | BC Type | Value |
|-------|---------|-------|
| `main_inlet` | `fixedValue` | **0** (pure CH4, no H2) |
| `branch_inlet` | `fixedValue` | **1** (pure H2) |
| `outlet` | `inletOutlet` | Zero-gradient on outflow; value 0 if backflow |
| `wall` | `zeroGradient` | No diffusive flux through walls |
| `sym` | `symmetryPlane` | Mirror |

### CH4 (Methane mass fraction)

Mirror image of H2: main_inlet = 1, branch_inlet = 0. The constraint H2 + CH4 = 1 holds everywhere (binary mixture, CH4 is the inert species).

### k (Turbulent kinetic energy)

| Patch | BC Type | Details |
|-------|---------|---------|
| `main_inlet` | `turbulentIntensityKineticEnergyInlet` | Turbulence intensity = 5%. k = 1.5 * (0.05 * U)^2. For U=10 m/s: k = 0.375 m^2/s^2. |
| `branch_inlet` | Same | Intensity = 5%, but U = U_branch (varies per case). |
| `wall` | `kqRWallFunction` | Standard wall function: uses log-law to compute k at the first cell. |
| `outlet` | `inletOutlet` | Zero-gradient outflow. |

### omega (Specific dissipation rate)

| Patch | BC Type | Details |
|-------|---------|---------|
| `main_inlet` | `turbulentMixingLengthFrequencyInlet` | Mixing length = 0.07 * D1 = 0.0322 m. omega = k^0.5 / (C_mu^0.25 * L_mix). |
| `branch_inlet` | Same | Mixing length = 0.07 * D2 (varies per case). |
| `wall` | `omegaWallFunction` | Blends viscous and log-law omega. |
| `outlet` | `inletOutlet` | Zero-gradient outflow. |

### nut (Turbulent viscosity)

Inlets/outlet: `calculated` (computed from k and omega by the turbulence model). Wall: `nutkWallFunction` (computes nut from k at near-wall cell).

### alphat (Turbulent thermal diffusivity)

All patches: `calculated` or `compressible::alphatWallFunction` (Prt = 0.85). Effectively inactive since energy equation is disabled.

## 6. Physical Properties

### Thermophysical model (`constant/thermophysicalProperties`)

| Setting | Choice | Meaning |
|---------|--------|---------|
| `type` | `heRhoThermo` | Enthalpy-based, density-based thermodynamics (compressible) |
| `mixture` | `reactingMixture` | Multi-species framework (needed for species transport) |
| `transport` | `sutherland` | Viscosity model: mu = As * sqrt(T) / (1 + Ts/T) |
| `thermo` | `janaf` | Heat capacity: 7-coefficient NASA polynomials Cp(T) |
| `energy` | `sensibleEnthalpy` | Energy equation in terms of sensible enthalpy h |
| `equationOfState` | `perfectGas` | Ideal gas law: rho = p*M / (R*T) |
| `specie` | `specie` | Standard species definition with molecular weight |

**Species data:**
- H2: molecular weight = 2.016 kg/kmol, Sutherland As = 6.69e-7, Ts = 72 K
- CH4: molecular weight = 16.043 kg/kmol, Sutherland As = 1.17e-6, Ts = 164 K

### Turbulence model (`constant/turbulenceProperties`)

- **k-omega SST** (Menter 1994): Industry standard. Blends k-omega (accurate near walls) with k-epsilon (accurate in freestream).
- **Sct = 0.5**: Turbulent Schmidt number. Controls turbulent species diffusion. Lower than the default 0.7 -- chosen to match experimental T-junction mixing data.
- kMin / omegaMin = 1e-10: Clipping floors to prevent division by zero.

### Gravity (`constant/g`)

- Direction: (0, -9.81, 0) m/s^2
- Gravity acts downward (-y). The branch enters from above (+y for top injection) or below (-y for bottom injection).
- This creates the buoyancy effects that drive H2 stratification.

### Combustion (`constant/combustionProperties`)

- `combustionModel none`: No combustion. The `reactions` file is empty. This is a cold mixing study only.

## 7. Solver Settings

### controlDict (Time control)

| Setting | Value | Why |
|---------|-------|-----|
| `application` | `rhoReactingBuoyantFoam` | Transient, compressible, buoyant, multi-species solver |
| `endTime` | 1.2 s | ~2 main-pipe flow-throughs (L/U = 6.9/10 = 0.69 s) |
| `deltaT` | 1e-5 s | Initial time step (adaptive) |
| `adjustTimeStep` | yes | Courant-based adaptive stepping |
| `maxCo` | 3.5 | Maximum Courant number (stability limit) |
| `writeInterval` | 0.1 s | Saves fields every 0.1 s |
| `purgeWrite` | 3 | Keeps only the 3 most recent time directories (saves disk) |

**Function objects** (run during the simulation):

1. **Residual monitoring**: Logs solver residuals for U, p_rgh, H2, k, omega every time step.
2. **y+ audit**: Writes y+ field every 0.1 s to verify wall-function validity (target: 30-200).
3. **Mass balance**: Monitors mass flux (phi) at main_inlet, branch_inlet, outlet. Verifies conservation.
4. **Pressure drop**: Area-averaged p_rgh at inlet and outlet every time step.
5. **Outlet H2**: Area-averaged H2 at outlet for CoV tracking.
6. **Downstream stations**: Area-averaged H2 and p_rgh on cutting planes at z = 3, 4, 5, 6 m (for CoV vs z/D plots).
7. **Field averaging**: Time-averages of U, H2, CH4, p_rgh, k starting at t = 0.6 s. Produces `H2Mean`, `UMean`, etc. Also computes variance (`H2Prime2Mean`) for segregation intensity.

### fvSchemes (Discretisation -- how PDEs become algebra)

| Equation part | Scheme | Order | Why |
|--------------|--------|-------|-----|
| Time derivative (ddt) | `Euler` | 1st order | Stable, monotonic convergence |
| Gradients | `Gauss linear` | 2nd order | Standard central differencing |
| Convection (div) | `upwind` | 1st order | Prioritises stability for DoE screening. Introduces ~5-10% numerical diffusion on CoV, but this is acceptable for a parametric sweep. |
| Diffusion (laplacian) | `Gauss linear corrected` | 2nd order | With non-orthogonal correction |
| Wall distance | `meshWave` | - | Needed for k-omega SST's blending function |

**Why first-order upwind for convection?** This is a deliberate trade-off. Second-order schemes (like `linearUpwind`) are more accurate but can produce oscillations (wiggles) near sharp gradients (the H2 jet core). Since we're running 40 cases as a screening DoE, robust convergence is more important than squeezing out the last 5% of accuracy. All cases are compared on equal footing.

### fvSolution (Linear solvers and algorithm)

**PIMPLE algorithm** (pressure-velocity coupling):

| Setting | Value | Meaning |
|---------|-------|---------|
| `nOuterCorrectors` | 1 | Effectively PISO mode (one outer loop per time step) |
| `nCorrectors` | 2 | Two pressure correction steps |
| `nNonOrthogonalCorrectors` | 1 | One non-orthogonal mesh correction |
| `momentumPredictor` | yes | Solve momentum before pressure |

**Linear solvers:**

| Field | Solver | Preconditioner | Tolerance |
|-------|--------|---------------|-----------|
| p_rgh | PCG | DIC | 1e-7 (relTol 0.01) |
| U | smoothSolver | symGaussSeidel | 1e-7 (relTol 0.1) |
| k, omega | PBiCGStab | DILU | 1e-7 (relTol 0.1) |
| H2, CH4 | PBiCGStab | DILU | 1e-7 (relTol 0.1) |
| Energy (h) | **maxIter = 0** | - | Disabled (isothermal) |

**Relaxation factors:**
- U: 0.7 (moderate under-relaxation)
- k, omega: 0.7
- H2, CH4: 0.9 (light relaxation -- species converge fast)
- p_rgh: 0.3 (heavy relaxation -- pressure needs it for stability)
- h: 0 (disabled)

## 8. Running the Simulation

### The Allrun script (per-case pipeline)

Each case has an `Allrun` script that executes the full pipeline in order:

```bash
#!/usr/bin/env bash

# 1. Source OpenFOAM environment
source /opt/OpenFOAM/OpenFOAM-v2406/etc/bashrc

# 2. Source per-case parameters

# 3. GEOMETRY: Generate STL files

python3 scripts/clip_stls.py    # Clip to half-domain (x >= 0)

# 4. MESHING (or reuse from donor case)
if [ -n "$REUSE_MESH_FROM" ]; then
    # Copy mesh from another case in the same slice
    cp -r "$REUSE_MESH_FROM/constant/polyMesh" constant/
else
    # Build mesh from scratch
    surfaceFeatureExtract        # Extract edge features from STLs
    blockMesh                    # Create background hex mesh
    snappyHexMesh -overwrite     # Refine + snap + add layers

# 5. DECOMPOSE for parallel run
decomposePar -force    # Split mesh into 16 subdomains

# 6. RUN THE SOLVER
mpirun -n 16 rhoReactingBuoyantFoam -parallel

# 7. POST-PROCESS
reconstructPar -latestTime    # Merge parallel results

# 8. Mark completion
touch SIM_DONE
```

## 9. Post-Processing and Metrics

### How CoV is computed (`all_metrics.py`)

CoV (Coefficient of Variation) measures mixing uniformity at the outlet:

```
CoV = sigma(Y_H2) / mean(Y_H2)
```

Where sigma is the standard deviation and mean is the average of H2 mass fraction across all outlet face cells.

**Detailed steps:**

1. Read the mesh: `constant/polyMesh/points`, `faces`, `boundary`
2. Identify the `outlet` patch (face indices and count from `boundary` file)
3. Read H2 field at the final time step (t = 1.2 s): `1.2/H2`
4. Extract H2 values on outlet faces
5. Compute face areas from the mesh geometry (cross-product of edge vectors)
6. Compute area-weighted mean: `mean = sum(H2_i * A_i) / sum(A_i)`
7. Compute area-weighted variance: `var = sum((H2_i - mean)^2 * A_i) / sum(A_i)`
8. CoV = sqrt(var) / mean

**Three weighting methods are computed:**
- Area-weighted (the primary metric)
- Mass-flux-weighted (accounts for velocity profile)
- Volume-flux-weighted

**CoV = 0** means perfect mixing (all outlet cells have identical H2 concentration).
**CoV = 1** means completely unmixed (binary: some cells pure H2, others pure CH4).

### How pressure drop is computed

The pressure drop is the difference between area-averaged pressure at the inlet and outlet:

```
|Î”P| = |<p_rgh>_inlet - <p_rgh>_outlet|
```

**The acoustic noise problem:**

The compressible solver (`rhoReactingBuoyantFoam`) captures acoustic pressure waves that bounce between the inlet (acoustically hard boundary) and outlet (acoustically soft boundary). These acoustic oscillations have amplitude ~25 kPa, while the actual flow-loss signal is only ~1-3 kPa.

**Solution: Low-pass filtering**

The `all_metrics.py` script applies a **causal moving-average filter** with width Î”tLP = 0.10 s (spanning ~3 acoustic round-trip periods). This suppresses the standing-wave content while preserving the steady-state pressure drop signal. The LP-filtered mean on the interval [0.4, 1.2] s is reported as the pressure drop.

### CoV vs z/D plots

The function objects in `controlDict` sample H2 at 4 downstream stations (z = 3, 4, 5, 6 m). Combined with the junction location (z = 2.3 m) and the pipe diameter (D = 0.46 m), this gives CoV at z/D = 1.5, 3.7, 5.9, 8.0, and at the outlet (~10 D). The CoV vs z/D plot shows how mixing improves with downstream distance.

### How the strip figure works (the griddata approach)

The strip figure shows 5 cross-sections of H2 in a row. The rendering uses:

1. **Slice the 3D mesh** at each z-station using PyVista
2. **Extract point data**: get (x, y, H2) coordinates from the slice
3. **Mirror**: reflect x < 0 data to x > 0 (since we only simulated the x >= 0 half)
4. **Interpolate onto a regular grid**: use `scipy.interpolate.griddata` with linear interpolation onto a 300x300 grid
5. **Mask outside the pipe**: set NaN for points where x^2 + y^2 > R_main^2
6. **Render with matplotlib**: `pcolormesh` with a shared colour scale across all 5 panels

This approach avoids the "mirror seam" artifact that occurs when trying to merge the reflected mesh halves in PyVista.

## 10. Design of Experiments (DoE)

### The sampling strategy: Sliced Latin Hypercube

**Why LHS?** Latin Hypercube Sampling ensures uniform coverage of the design space with fewer points than a full factorial grid. Each variable's range is divided into N equal strata, with exactly one sample per stratum.

**Why "sliced"?** The Ye-Li-Sudjianto (2000) sliced LHS divides the d/D axis into 5 bins (slices). Within each slice, d/D is fixed, and 2 HBR values are sampled. This gives:

- 5 unique d/D values x 2 HBR values = **10 cases per campaign**
- Cases in the same slice share the same geometry -> can reuse the mesh
- 5 mesh builds instead of 10 -> halves meshing time

### The design table (10-case)

| Case | d/D | HBR (%) | VR | Branch velocity (m/s) |
|------|------|---------|------|----------------------|
| 01 | 0.196 | 18.4 | 5.84 | 58.4 |
| 02 | 0.196 | 6.0 | 1.64 | 16.4 |
| 03 | 0.252 | 7.8 | 1.33 | 13.3 |
| 04 | 0.252 | 19.5 | 3.81 | 38.1 |
| 05 | 0.296 | 9.8 | 1.24 | 12.4 |
| 06 | 0.296 | 18.7 | 2.61 | 26.1 |
| 07 | 0.382 | 14.2 | 1.14 | 11.4 |
| 08 | 0.382 | 9.2 | 0.69 | 6.9 |
| 09 | 0.396 | 13.0 | 0.95 | 9.5 |
| 10 | 0.396 | 11.2 | 0.81 | 8.1 |

VR is derived: `VR = HBR / ((1 - HBR) * (d/D)^2)`

### The same design is used across all 4 campaigns

The same 10 (d/D, HBR) pairs are used for all 4 campaigns (90 deg top, 30 deg top, 90 deg bottom, 30 deg bottom). This creates **matched pairs**: case 06 at 90 deg and case 06 at 30 deg have identical d/D, HBR, and VR -- only the angle differs. This enables rigorous paired-comparison statistical tests.

### The stamp_cases.py workflow

```
lhs_design.py -> doe_design.csv -> stamp_cases.py -> case_01/ ... case_10/
```

For each row in the CSV:
1. Copy `0/`, `constant/`, `system/`, `generateSTL.py` from the template
2. Replace `@TOKEN@` placeholders in `0/U`, `0/k`, `0/omega` with per-case values
3. Update `snappyHexMeshDict` refinement regions to match the per-case branch diameter
4. Generate `case.env`, `case_info.json`, `Allrun`

### The run_doe.sh orchestrator

Features:
- **Sequential execution**: runs cases one by one
- **Resumable**: skips cases with `SIM_DONE` sentinel. Use `--from 05` to resume from case 5
- **Mesh reuse**: automatically finds a donor case in the same slice
- **Resilience**: if a case fails, it logs the error, marks `SIM_FAILED`, and continues to the next
- **Post-processing**: after each case, runs `all_metrics.py`, `make_figures.py`, `make_distance_figures.py`
- **Summary**: after all cases, runs `make_doe_summary.py` for campaign-level scatter plots and heatmaps

## 11. Cross-Campaign Analysis

### The script: `cross_analysis.py`

After all 4 campaigns are complete (40 cases), this script combines the results for comparative analysis.

### Key analysis steps

1. **Load data**: reads `doe_summary.csv` from each campaign directory
2. **Recompute LP-filtered Î”P**: applies 100 ms moving-average to raw pressure time series for apples-to-apples comparison
3. **Matched-pair comparison**: for each design point, compares 90 deg vs 30 deg (or top vs bottom), computing percent change in CoV and Î”P
4. **Power-law fits**: fits CoV = A * VR^b in log-log space for each campaign
5. **Statistical tests**: binomial test for sign consistency (do all pairs improve?)
6. **Pareto front**: identifies non-dominated points in (CoV, |Î”P|) space

### Key findings

- **30 deg top injection wins**: median CoV 46% lower, median |Î”P| 63% lower than 90 deg

## 12. Surrogate Modelling

### The script: `train_surrogate.py`

Trains ML models on the 40-case CFD dataset to enable rapid optimisation without running more simulations.

### Features (inputs to the ML model)

6 features: `[side_top, side_bot, angle, d/D, VR, HBR]`
- `side_top` / `side_bot`: one-hot encoding (1/0) for injection side
- `angle`: injection angle in degrees
- All features are MinMax-scaled to [0, 1]

### Targets (outputs)

2 targets: `[CoV, |Î”P| (kPa)]`
- Both MinMax-scaled for training

### Models benchmarked

| Model | Architecture | CoV R^2 | CoV MAE |
|-------|-------------|---------|---------|
| **GPR** (best) | Gaussian Process, Matern-5/2 + White kernel | **0.925** | **0.083** |
| Gradient Boosting | 200 trees, lr=0.05, depth=3 | 0.897 | 0.096 |
| Random Forest | 200 trees, min_leaf=2 | 0.865 | 0.103 |
| DNN | 6-64-128-64-32-2, Adam, early stopping | 0.735 | 0.129 |

**Î”P prediction**: All models fail (R^2 ~ 0) because the acoustic noise (~25 kPa) completely masks the ~1-3 kPa flow-loss signal.

**Why GPR wins**: With only 40 training points, the Gaussian Process's Bayesian framework provides excellent uncertainty quantification and avoids overfitting. The DNN (17,400 parameters) severely overfits at N=40.

### NSGA-II optimisation

The best surrogate (GPR) is used inside a multi-objective genetic algorithm:
- **Algorithm**: NSGA-II (Non-dominated Sorting Genetic Algorithm II)
- **Population**: 200, **Generations**: 150
- **Crossover**: Simulated Binary (SBX), **Mutation**: Polynomial
- **Objectives**: minimise CoV AND minimise |Î”P|
- **Library**: pymoo

**Knee-point designs** (best trade-off):
- **Top injection**: Î± = 30 deg, d/D = 0.45, VR = 5.82 -> CoV ~ 0.089, |Î”P| ~ 0.14 kPa
- **Bottom injection**: Î± = 90 deg, d/D = 0.26, VR = 5.84 -> CoV ~ 0.048, |Î”P| ~ 0.97 kPa

### SHAP analysis

Kernel SHAP on the GPR model reveals feature importance:
- **Injection side** and **VR** are the two dominant features for CoV
- **Angle** is the third most important

## 15. How to Know Simulations Are Correct

### Checklist for each case

| Check | What to look for | Where to find it |
|-------|-----------------|-----------------|
| **Residuals** | All residuals < 1e-8 by end of simulation | `log.solver` (search for "Solving for") |
| **Continuity error** | Cumulative < 1e-2 | `log.solver` (search for "continuity errors") |
| **Mass balance** | < 0.5% deviation | `all_metrics.csv` or `postProcessing/` flux files |
| **No NaN/FPE** | No "nan" or "Floating Point Exception" in log | `log.solver` |
| **Courant number** | Mean Co < 1, max Co < 3.5 | `log.solver` |
| **Mesh quality** | checkMesh passes (max non-ortho < 65 deg) | `log.checkMesh` |
| **y+ values** | 30-200 range for wall functions | `postProcessing/yPlus/` |
| **H2 bounds** | 0 <= H2 <= 1 everywhere | Visual check in figures |
| **SIM_DONE sentinel** | File exists | `case_XX/SIM_DONE` |

### Grid independence study

Performed at one design point (case 03, 90 deg top):

| Mesh | Cells | CoV | |Î”P| | Change from previous |
|------|-------|-----|------|---------------------|
| Coarse | ~230k | 0.478 | 1.18 kPa | - |
| **Medium** | **~460k** | **0.472** | **1.16 kPa** | **CoV: 0.4%, Î”P: 1.6%** |
| Fine | ~920k | 0.471 | 1.15 kPa | CoV: 0.2%, Î”P: 0.7% |

The medium mesh (460k) was chosen because:
- Changes < 2% for both CoV and Î”P compared to the fine mesh
- Runs in ~40 minutes vs ~3 hours for the fine mesh
- Same baseline mesh is used for all 40 cases

### Signs that something went wrong

- **Divergence**: Courant number explodes, solver crashes with "FOAM FATAL ERROR"
- **2-region mesh**: `checkMesh` reports 2 or more regions (mesh is disconnected). The branch inlet may not be connected to the main pipe.
- **Frozen cells**: Some cells stay at H2 = 1.0 (initial value) through the entire simulation. These are orphan cells isolated by snappyHexMesh.
- **Acoustic blowup**: Pressure oscillates wildly (> 100 kPa amplitude). Usually means the time step is too large or boundary conditions are inconsistent.
